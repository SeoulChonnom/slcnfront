# 신규 디자인 개선/추가 필요 사항 확장 보고서

## 1. 문서 목적

이 문서는 현재 Vue 기반으로 구현된 SLCN 프로젝트를 기준으로,

- 기존 기능/정보구조/실제 화면
- 신규 PC 디자인(Stitch 산출물)
- 신규 Mobile 디자인(Pencil 산출물)
- React 기반 프론트엔드 전환 계획

을 함께 고려하여, **신규 디자인에서 추가 보완이 필요한 부분**, **기능 보존을 위해 수정해야 하는 부분**, **React 전환 시 구조적으로 다시 설계해야 하는 부분**을 정리한 확장판 보고서다.

본 문서는 단순 시각 비교가 아니라, "현행 기능을 잃지 않으면서 React 기반 신규 UI로 안전하게 이전할 수 있는가"를 중심으로 작성한다.

### 2.2 사용자 확인 반영 사항

아래는 사용자 답변을 반영해 본 보고서에서 확정 기준으로 사용한 내용이다.

1. `New Log`는 구현 대상으로 보되, 명칭은 **"새 나들이 기록하기"** 로 변경해야 한다.
2. `MyPage`, `Record`, `Membership`, `Profile`은 **이번 구현 범위에서 제외**한다.
3. 모바일 `주간 일정` 화면은 **현재 구현 범위에 포함**한다.
   - 단, React 기반 캘린더 라이브러리 선정이 어렵다면 제외 가능성이 있다.
4. 나들이 등록에서는 아래 필드를 계속 유지해야 한다.
   - `type`
   - `button1/button2`
   - `quizAnswerTitle/Text`
   - `quizErrorTitle/Text`
5. 드라이브 비밀번호 안내는 나들이 공통 정책이므로, **등록 페이지 입력 항목으로 받을 필요는 없다.**
6. 신발 추천은 `src/global/global.js`의 `globalShoes` 데이터를 유지하며, 이 데이터를 기반으로 화면을 구성한다.

## 3. 전체 결론

신규 디자인은 전반적으로 올바른 방향이다. 특히 아래 변화는 유지할 가치가 높다.

- 단순 카드형 메인에서 브랜딩 강화형 홈으로의 개선
- 모바일 Bottom Navigation 도입
- 목록/상세의 정보 밀도 강화
- 나들이 등록의 단계형 폼 전환
- 신발 추천의 카탈로그/에디토리얼 구조 강화

다만 현재 최종 디자인은 **"구현 직전 완성본"이라기보다, 기능 보존을 위해 추가 보정이 필요한 최종안"** 에 가깝다. 핵심 이유는 다음과 같다.

1. `/map/register`는 실제 백엔드/현재 기능 요구사항을 충분히 담고 있지 않다.
2. 일부 디자인 메뉴는 구현 범위 밖 기능이 섞여 있어 IA 정리가 필요하다.
3. 모바일 주간 일정은 구현 범위에 포함되지만, React 전환과 라이브러리 선택에 따라 실현 가능성이 달라진다.
4. 신발 추천은 디자인보다 실제 데이터(`globalShoes`)를 우선해야 한다.

즉, 이번 작업은 단순 스타일 변경이 아니라 **기능 보존형 UI 리디자인 + 프레임워크 전환 설계**로 보는 것이 맞다.

## 4. 최우선 수정 필요 항목

### P1. 나들이 등록 `/map/register`는 디자인 보완 없이는 구현하면 안 됨

현재 Vue 구현 기준 `mapRegisterPage.vue`는 다음 데이터를 처리한다.

- `type`
- `date`
- `logo`
- `info2` (나들이 이름)
- `map1`
- `map2` (선택)
- `button1`
- `button2`
- `drive`
- `quizTitle`
- `quizAnswer`
- `quizAnswerTitle`
- `quizAnswerText`
- `quizErrorTitle`
- `quizErrorText`
- `quizRegisterRequestList`

현재 신규 디자인은 Step 구조는 좋지만, 실질적으로 아래가 비거나 약하다.

- `type` 선택
- 복수 지도 시 `button1/button2`
- 정답 선택 UI
- 정답 모달용 문구
- 오답 모달용 문구
- `drive` 링크 입력의 존재감

반면 아래는 확정적으로 **입력 항목에서 제외 가능**하다.

- 드라이브 비밀번호 안내

권고:

- 등록 화면은 다음과 같이 다시 정리해야 한다.
  - STEP 1: 기본 정보
    - `type`
    - 날짜
    - 나들이 이름
    - 로고 업로드
  - STEP 2: 경로/지도
    - `map1`
    - `map2` 토글
    - `button1/button2`
    - `drive`
  - STEP 3: 퀴즈 설정
    - `quizTitle`
    - 보기 4개
    - `quizAnswer`
    - `quizAnswerTitle/Text`
    - `quizErrorTitle/Text`
- Mobile 최종본의 `OPTION A~D` 구성은 유지 가능하지만, 반드시 "정답 선택" UI를 별도 추가해야 한다.
- 현재 시안의 간결함은 유지하되, 데이터 구조를 숨기지 않는 방향으로 재설계해야 한다.

### P2. 공통 IA에서 미구현 메뉴 제거 또는 숨김 처리 필요

이번 범위에서 제외되는 메뉴:

- `MyPage`
- `Record`
- `Membership`
- `Profile`

이번 범위에서 유지되는 신규 요소:

- `New Log` → 명칭을 **"새 나들이 기록하기"** 로 변경 후 유지

문제:

- 현재 PC 시안 헤더/카드 곳곳에 범위 밖 메뉴가 섞여 있으면 React 구현 시 라우트 미정 링크가 생긴다.
- 이는 디자인 완성도를 떨어뜨릴 뿐 아니라, App Shell 설계도 불안정하게 만든다.

권고:

- 공통 메뉴는 이번 범위에서 아래만 유지하는 것이 맞다.
  - 홈
  - 나들이 기록
  - 일정
  - 신발 추천
  - Choi's Film Art
  - 새 나들이 기록하기
- `새 나들이 기록하기`는 전역 메뉴가 아니라, 맥락형 CTA로 두는 편이 더 적절하다.
  - 메인 카드
  - 나들이 목록 하단 고정 버튼
  - 필요 시 헤더 우측 액션 버튼

### P3. 모바일 주간 일정은 "디자인 확정"과 "기술 가능성"을 함께 판단해야 함

주간 일정 화면은 이번 구현 범위에 포함되었다. 따라서 단순 아이디어가 아니라 **실제 개발 가능 여부 검토 대상**이다.

하지만 현재 문제는 다음과 같다.

- Vue 구현은 TOAST UI Calendar 기반의 월간 CRUD다.
- React 전환 시 기존 달력 구현을 그대로 가져가기 어렵다.
- 모바일 주간 뷰를 잘 지원하는 React 캘린더 라이브러리가 제한적일 수 있다.

권고:

- 주간 일정은 디자인상 유지하되, 기술 전략을 먼저 정해야 한다.
- 우선순위는 아래가 현실적이다.
  1. React 라이브러리로 월간/주간 모두 지원 가능한지 검토
  2. 가능하면 하나의 라이브러리로 월간/주간 통합
  3. 불가능하면 월간 우선, 주간은 커스텀 뷰 또는 범위 제외 판단

즉, 주간 일정은 지금 단계에서 "반드시 구현"이라고 문서에 박기보다, **기술 검증 후 포함/제외가 갈릴 수 있는 조건부 범위**로 쓰는 것이 안전하다.

### P4. 신발 추천은 디자인보다 `globalShoes` 데이터가 기준이어야 함

이 부분은 사용자 답변으로 기준이 명확해졌다.

- 콘텐츠 기준: `src/global/global.js`
- 브랜드 기준: 뉴발란스 / 나이키 / 아식스
- 상세 기준: 설명, 영상 유무, 리뷰 이미지 2개, 외부 링크

따라서 신규 디자인은 구조 참고용이지, 상품 수/텍스트/가격/브랜드 구성을 새로 정의하는 기준이 아니다.

권고:

- PC/모바일 모두 `globalShoes` 기준으로 레이아웃을 맞춘다.
- 모바일 카탈로그에는 반드시 `아식스` 섹션을 추가한다.
- 상세 페이지는 영상 존재 여부에 따라 블록이 조건부 렌더링되어야 한다.

## 5. 화면별 상세 분석

### 5.1 로그인 `/login`

현재 기능:

- 입력 로그인
- 리프레시 토큰 기반 자동 로그인

신규 디자인 평가:

- 브랜딩 강화는 적절하다.
- Mobile 로그인 화면도 현재 기능과 구조 충돌은 없다.

추가 보완 필요:

- 자동 로그인 시 잠깐의 로딩 상태가 필요하다.
- 인라인 에러가 없는 현재 alert 방식은 신규 UI와 어울리지 않는다.
- 한국어 서비스 기준 텍스트 통일 필요
- 접근성 연결(`label`/`id`)은 React 전환 시 정리해야 한다.

React 전환 권고:

- 로그인 폼은 `react-hook-form + zod` 조합이 적합하다.
- 서버 호출은 `TanStack Query mutation` 또는 별도 auth action으로 분리하는 편이 좋다.
- silent login은 라우트 진입 시 세션 복원 로직으로 분리하는 편이 구조적으로 낫다.

### 5.2 메인 대시보드 `/`

현재 기능:

- D-day
- 달력 이동
- 나들이 기록 이동
- Choi's Film Art 외부 링크
- 신발 추천 이동

신규 디자인 평가:

- 기존보다 훨씬 좋은 방향이다.
- 특히 모바일 메인 카드 구조와 BottomNav 도입은 유지 가치가 높다.

보완 필요:

- `New Log`는 명칭을 **"새 나들이 기록하기"** 로 바꿔야 한다.
- 미구현 메뉴는 제거해야 한다.
- D-day 카드의 날짜/문구는 실데이터 길이 기준 검증이 필요하다.
- 최근 나들이 프리뷰를 넣는다면 empty state도 필요하다.

React 전환 권고:

- 메인은 `AppShell + FeatureCard + SectionBlock` 조합으로 분리하는 것이 적절하다.
- D-day는 하드코딩보다 설정값 또는 서버 데이터로 분리하는 편이 유지보수에 좋다.

### 5.3 나들이 목록 `/map`

현재 기능:

- 목록 조회
- 카드 클릭 시 퀴즈
- 정답일 때 상세 이동
- 등록 진입

신규 디자인 평가:

- 검색바 도입은 유효한 개선이다.
- 카드 정보량 확장은 방향이 좋다.
- 모바일 하단 CTA는 특히 적합하다.

보완 필요:

- 퀴즈 통과 후 상세 진입이라는 규칙이 UI에서 더 드러나야 한다.
- 검색이 들어간다면 검색 기준 정의가 필요하다.
- 목록 empty/error/loading 상태가 디자인에 추가되어야 한다.
- "새 나들이 기록하기"는 모바일뿐 아니라 PC에서도 시야에 잘 들어와야 한다.

React 전환 권고:

- 목록/검색/퀴즈 모달 흐름은 도메인 단위로 분리하는 편이 좋다.
- 예시 구조
  - `OutingListPage`
  - `OutingCard`
  - `OutingSearchBar`
  - `QuizUnlockModal`
- 서버 목록은 `TanStack Query`로 관리하는 것이 적합하다.

### 5.4 나들이 등록 `/map/register`

현재 기능:

- multipart 업로드
- 단일/복수 지도
- 퀴즈 설정
- 각종 정답/오답 문구 입력

신규 디자인 평가:

- 긴 단일 폼을 단계형으로 바꾸는 방향은 매우 좋다.
- 모바일 최종본도 기본적인 정보 구조는 괜찮다.

핵심 문제:

- 실제 입력 구조가 시안보다 훨씬 복잡하다.
- 지금 시안 그대로 구현하면 기능 유실 가능성이 높다.

React 전환 권고:

- 이 화면은 React 전환 시 가장 재설계 가치가 높은 페이지다.
- 권장 스택
  - `react-hook-form`
  - `zod`
  - step별 validation schema
  - 파일 업로드 상태 관리 분리
- 이 페이지는 공통 입력 컴포넌트의 기준 화면으로 활용하기 좋다.

### 5.5 나들이 상세 `/map/:date`

현재 기능:

- 지도 이미지 표시
- 멀티맵 토글
- 드라이브 링크
- 공통 비밀번호 안내

신규 디자인 평가:

- 세그먼트형 지도 전환은 기존보다 낫다.
- 하단 CTA 고정도 매우 적절하다.

보완 필요:

- 세그먼트 라벨은 예시 지명이 아니라 `button1/button2` 기반 동적 데이터여야 한다.
- 공통 비밀번호 안내는 등록 입력 필드는 아니지만, 상세 화면에서는 유지 검토가 필요하다.
- 목록 복귀 동선은 더 명시적으로 설계하는 것이 좋다.

React 전환 권고:

- 상세는 `query + route param` 구조로 구현하면 된다.
- 멀티맵 전환은 local state로 충분하다.
- 이미지 로딩/실패/재시도 상태를 명확히 넣는 것이 좋다.

### 5.6 일정 관리 `/calendar`

현재 기능:

- 월간 뷰
- 생성/수정/삭제
- TOAST UI Popup 사용

신규 디자인 평가:

- 시각적 완성도는 확실히 올라간다.
- 모바일 월간/주간 화면 분리는 UX 측면에서 납득 가능하다.

보완 필요:

- 월간과 주간이 모두 구현 범위인 만큼, 단순 시안 수준이 아니라 실제 동작 흐름까지 설계되어야 한다.
- 일정 생성/수정/삭제가 어디서 어떻게 발생하는지 최종 디자인 문서에 더 구체화할 필요가 있다.

React 전환 권고:

- 이 영역은 프레임워크 전환 리스크가 가장 큰 영역 중 하나다.
- 판단 기준
  - 월간/주간 모두 자연스럽게 지원하는 React 라이브러리가 있는가
  - 기존 CRUD API와 쉽게 연결되는가
  - 모바일 주간 UI를 커스터마이즈 가능한가
- 만약 적합한 라이브러리가 없으면
  - 월간은 라이브러리
  - 주간은 커스텀 구현
  - 또는 주간 범위 조정
    로 나누는 전략이 필요하다.

### 5.7 신발 추천 목록 `/shoesRecom`

현재 기능:

- 정적 브랜드/상품 카탈로그
- 상세 진입

신규 디자인 평가:

- 카탈로그 구조는 유지 가치가 높다.
- 카드형 정렬은 실제 데이터와도 잘 맞는다.

보완 필요:

- 모바일에서 `아식스` 추가
- 실제 `globalShoes` 텍스트 길이 기준으로 카드 높이 확인
- 브랜드별 상품 수 차이에 대응할 레이아웃 규칙 필요

React 전환 권고:

- 서버 상태가 아니므로 단순 static domain으로 관리 가능하다.
- 다만 향후 API 전환 가능성을 고려해 data adapter layer를 두는 것이 좋다.

### 5.8 신발 상세 `/:brand/:shoesName`

현재 기능:

- 상품 이미지
- 설명
- 조건부 영상
- 리뷰 이미지 2개
- 외부 링크

신규 디자인 평가:

- 에디토리얼형 상세 구성은 적절하다.
- 리뷰 카드 구조도 데이터와 잘 맞는다.

보완 필요:

- 영상 없는 상품에서 레이아웃 처리 기준이 필요하다.
- 잘못된 URL 접근 시 404/복귀 UX가 필요하다.
- 외부 링크 클릭 affordance를 강화할 필요가 있다.

React 전환 권고:

- route param validation을 명확히 넣는 것이 좋다.
- 존재하지 않는 `brand/shoesName` 조합은 404 또는 목록 복귀 화면으로 유도해야 한다.

## 6. React 프레임워크 전환 관점에서의 설계 제안

### 6.1 전환 관점 핵심 판단

이번 프로젝트는 단순 Vue → React 포팅이 아니다. 실제로는 아래 세 가지를 동시에 수행한다.

1. UI/UX 리디자인
2. 공통 App Shell 재설계
3. 상태/폼/라우팅 구조 재편

따라서 파일 단위 1:1 치환보다, **도메인 단위 재구성**이 더 적합하다.

### 6.2 권장 구조

예시 구조:

- `app/`
  - 라우터
  - 전역 providers
  - theme
- `features/auth/`
- `features/outing/`
  - list
  - detail
  - register
  - quiz
- `features/calendar/`
  - month
  - week
  - event-editor
- `features/shoes/`
  - catalog
  - detail
- `components/ui/`
  - 버튼
  - 입력
  - 배지
  - 시트
  - 모달
  - AppShell

### 6.3 상태 관리 제안

- 서버 상태: `TanStack Query`
- 폼 상태: `react-hook-form`
- 검증: `zod`
- 가벼운 UI 상태:
  - 모달
  - 토글
  - 탭
  - 임시 step 상태
    는 React local state 또는 소형 store로 충분하다.

### 6.4 라우팅 제안

현재 라우트 구조는 유지 가능하다.

- `/`
- `/login`
- `/map`
- `/map/register`
- `/map/:date`
- `/calendar`
- `/shoesRecom`
- `/:brand/:shoesName`

다만 주의점:

- `/:brand/:shoesName`는 범용 동적 라우트라 충돌 가능성을 항상 점검해야 한다.
- React Router 기준에서는 정적 경로 우선 배치와 404 처리 설계가 중요하다.

### 6.5 App Shell 제안

React 전환 시 가장 먼저 잡아야 할 것은 공통 레이아웃이다.

권장 규칙:

- 로그인: App Shell 최소화
- 메인/목록/캘린더/신발목록: TopAppBar 또는 Header + BottomNav
- 상세/등록: 뒤로가기 우선, BottomNav 숨김 또는 조건부 처리

이 규칙이 먼저 정리되어야 페이지 구현이 안정된다.

## 8. 확정 반영이 필요한 디자인 수정안

아래 항목은 이번 답변 기준으로 **거의 확정적으로 반영해도 되는 수정안**이다.

1. `New Log` → **"새 나들이 기록하기"**
2. `MyPage`, `Record`, `Membership`, `Profile` 제거 또는 비노출
3. 나들이 등록 시안에 `type`, `button1/button2`, `quizAnswerTitle/Text`, `quizErrorTitle/Text`, `quizAnswer` 추가
4. 등록 시안에서 드라이브 비밀번호 입력 항목 제거
5. 모바일 신발 추천에 `아식스` 섹션 추가
6. 신발 추천/상세는 `globalShoes` 기반으로 UI 길이와 구조 재검증
7. 캘린더 주간 뷰는 기술 검증 전제 조건부 범위로 표시

## 10. 최종 제안

이번 프로젝트는 "기존 Vue 서비스를 React로 옮기면서, 신규 디자인을 기능 손실 없이 재정렬하는 작업"으로 보는 것이 가장 정확하다.

따라서 최종 방향은 아래가 적절하다.

- IA는 **현행 기능 기준으로 정리**
- 신규 디자인은 **시각 방향과 레이아웃 패턴을 적극 수용**
- 단, 등록/캘린더/공통 메뉴는 **기능 기준으로 다시 보정**
- React 전환은 **페이지 복제보다 도메인 재설계 방식**으로 진행

핵심적으로는 다음 한 문장으로 정리할 수 있다.

**신규 디자인은 채택 가능하지만, 현재 기능과 React 전환 조건에 맞춰 구조 보정이 선행되어야 한다.**
